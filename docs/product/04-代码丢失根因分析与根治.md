## 2. 调查过程与证据

### 2.1 本地 git 状态

```
分支: feature/m0-mvp (7f185e3) ← 仅"克隆项目"commit
      main           (a5b0860) ← 仅"90 天文案"commit

reflog（仅 4 条，全是 clone/checkout）:
  7f185e3 HEAD@{0}: checkout: moving from main to feature/m0-mvp
  a5b0860 HEAD@{1}: clone: from https://github.com/hpj360/Hermes

git fsck --unreachable: （无悬挂对象）
git stash list: （空）
git cat-file -t 1b61eea: fatal: Not a valid object name
```

**结论 1**：本地 git 对象库中**根本没有** `1b61eea` 这个 commit，本地无任何残留可恢复。

### 2.2 远端 git 状态

执行 `git fetch '+refs/heads/*:refs/remotes/origin/*' '+refs/pull/*:refs/remotes/origin/pr/*'` 后检查所有远端分支：

| 远端分支 | HEAD | 含 hermes_kb | 含 test_m1 | 含产品文档 |
|---|---|---|---|---|
| `origin/main` | a5b0860 | ❌ | ❌ | ❌ |
| `origin/feature/m0-mvp` | 7f185e3 | ❌ | ❌ | ❌ |
| `origin/feature/dev` | 523e5f6 | ❌ | ❌ | ❌ |
| `origin/trae/agent-7jQWQ1` | fb4403b | ❌ | ❌ | ❌ |
| `origin/trae/agent-glOxQF` | d17a4b0 | ❌ | ❌ | ❌ |
| `origin/pr/1/head` | d17a4b0 | ❌ | ❌ | ❌ |
| `origin/pr/1/merge` | 295fc2b | ❌ | ❌ | ❌ |

**结论 2**：远端 GitHub **所有 6 个分支 + 2 个 PR ref** 均无 M0/M1 代码、无 test_m1、无产品文档。PR #1 是无关项目（loop-engineering / douyin-reader），与酒类知识库无关。

### 2.3 clone 配置缺陷

```
.git/config:
  [remote "origin"]
    url = https://github.com/hpj360/Hermes
    fetch = +refs/heads/main:refs/remotes/origin/main   ← 仅拉 main！
```

`git clone` 默认应拉取所有分支，但此仓库被人为限制为只 fetch main。导致：
- 即便远端其他分支有 M0/M1 代码，本地也看不到
- 调查时一开始误以为"远端只有 main 分支"

**结论 3**：clone 配置仅拉 main 是次因，但**不是**代码丢失主因（远端确实没有 M1 代码）。

## 3. 根本原因

### 3.1 直接原因
**前序会话的 commit `1b61eea` 从未推送到 GitHub 远端。**

证据：远端所有分支均无此 commit，远端 SHA 与本地不一致。

### 3.2 根本原因（5 Why 分析）

```
Why 1: 为什么代码丢失？
  → 工作区被重置（重新 clone），本地 git 历史清空。

Why 2: 为什么重置会丢失？
  → 因为 commit 仅存在于本地临时沙箱文件系统，未推送到远端持久化。

Why 3: 为什么没推送到远端？
  → 前序会话执行了 git commit 但未执行 git push，或 push 失败未察觉。

Why 4: 为什么没 push / push 失败未察觉？
  → 缺乏"commit 后自动 push"的强制机制；缺乏"会话结束前校验推送"的流程。

Why 5: 为什么流程缺失？
  → 依赖人工记忆执行 push，未建立工具级防护（hook/脚本/CI 校验）。
```

### 3.3 促成因素

| 因素 | 影响 |
|---|---|
| **沙箱生命周期短** | 远程沙箱每次会话独立，会话结束文件系统清空，本地 commit 随之消失 |
| **无自动 push hook** | 标准 git 流程 commit 后 push 是手动步骤，容易遗漏 |
| **无会话结束校验** | 没有在收尾时强制校验"本地 commit 是否已推送" |
| **fetch 配置错误** | `remote.origin.fetch` 仅拉 main，掩盖了其他分支的存在，调查初期误导 |
| **缺乏工作完成 checklist** | 没有"代码已推送"的硬性验收项 |
| **会话间无状态传递** | 前序会话的"已完成"状态无法验证，仅靠总结文字记录 |

### 3.4 根因分类

```
              ┌─ 直接原因
              │   └─ commit 未 push 到远端
              │
根因 ──────────┼─ 流程缺陷（主因）
              │   ├─ 缺少 commit→push 自动化
              │   ├─ 缺少会话结束校验
              │   └─ 缺少工作完成 checklist
              │
              └─ 环境因素（次因）
                  ├─ 沙箱会话隔离
                  └─ fetch 配置仅拉 main
```

## 5. 使用规范

### 5.1 会话开始时

```bash
./scripts/preflight.sh
```

预检内容：
- fetch 远端最新状态
- 比较本地与远端 SHA（领先/落后/分叉）
- 检查未提交改动
- 校验关键目录存在（防止重置后裸奔）

### 5.2 提交代码时（推荐）

```bash
./scripts/safe_commit.sh "feat: 描述"
```

替代手动 `git add + commit + push`，自动完成：
1. `git add -A`
2. `git commit -m "..."`
3. `git push`（首次自动 `-u`）
4. 校验本地 SHA = 远端 SHA

### 5.3 会话结束前（强制）

```bash
./scripts/verify_pushed.sh
```

校验：
- 本地无未推送 commit
- 工作区无未提交改动

**退出码非 0 时，禁止结束会话。**

### 5.4 普通 commit 流程

即便用 `git commit` 而非 `safe_commit.sh`，post-commit hook 也会自动 push。失败时输出醒目警告。

## 8. 教训与流程改进

### 8.1 核心教训

1. **commit ≠ 保存**：本地 commit 在沙箱重置后会消失，必须 push 到远端才算"保存"
2. **不要依赖人工记忆**：push 应该是自动化的，不应该是"记得做"的事
3. **会话切换是高风险点**：每次会话开始应预检，结束应校验
4. **fetch 配置影响调查方向**：错误的 fetch 配置会掩盖真相

### 8.2 流程改进

| 改进项 | 旧流程 | 新流程 |
|---|---|---|
| 提交代码 | `git add + commit`（手动 push） | `./scripts/safe_commit.sh`（自动 push） |
| 会话开始 | 直接开干 | `./scripts/preflight.sh` 预检 |
| 会话结束 | 直接结束 | `./scripts/verify_pushed.sh` 校验 |
| 远端配置 | 仅 fetch main | fetch 全分支 + PR |
| 状态确认 | 凭记忆/总结 | git SHA 比对 |

### 8.3 工作完成 Definition of Done（DoD）

代码任务"完成"必须同时满足：
- [ ] 所有改动已 commit
- [ ] commit 已推送到远端（`git log @{u}..` 为空）
- [ ] `./scripts/verify_pushed.sh` 退出码 0
- [ ] 关键测试通过（`pytest tests/`）
- [ ] 涉及前端时 `npm run build` 成功
